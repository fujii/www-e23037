仕様

  1. emacs-w3mのようにEmacs内で使用
  2. html や 画像はダウンロードしながら表示する。 こうでなくては、体感速度が遅くなる。
  3. レンダリングはCで、入力関係はelispで実装。
  4. table、frameのレンダリング。 w3mを使ったらlynxには戻れない。
  5. css対応。lynxやemacs/W3がやってるし。
  6. XML対応。今後のことを考えて。
  7. カスタマイズできる。とくにキーバインドはもはや宗教。 あと、画像のダウンロードもgaleonのように。 もちろん、カスタマイズはelispにて。
  8. JS(JavaScript)対応
  9. できるかぎりプロポーショナルフォント対応。 罫線(┨ ┯ ┗ ├ など)でtable枠を書くため、原理的に不可能でない範囲で。
  10. ページのキャッシュはEmacsのバッファとしてではなく、 C側で蓄えた方がいいのかな。
  11. Emacs21とXEmacs21がターゲットかも。
  12. 文字コーディングの指定とコーディングの自動判別の指定は別物。
  13. SVGは特許について調べてから考える。

つまり、emacs/W3をCで再発明。

利用できそうなもの

  • w3m とそのvariant
  • emacs-w3m
  • lynx

  • NJS JavaScript Interpreter

Debianも使う者としてはライセンスのチャンポンは避けるべきなのだろうか？ GPLとLGPLしか利用しないとなると、作業量は増えそうだ。

今後の予定

Cとelispの通信プロトコルをどうしよう。 JS対応となると複雑そう。

Cの内部文字コードは何にしよう。 UCSより、Emacsの内部文字コードと同じにした方が高速になりそう。 それって、プロトコルに使えるのだろうか？

仕様が決まれば、利用するfree softwareの選定とソース読み。

協力者募集中。fujii@chi.its.hiroshima-cu.ac.jp まで。

Emacsの内部文字コードはemacs-muleということは、 調べてみるとすぐにわかった。 しかし、内部コードをプロトコルに使えるのだろうか？ あと、font-lockを含めた内部コードは何なんだろう？ また、emacs-muleということはmule機能と関係してそうな名前だが、 英語圏の人はmule機能を使ってるのだろうか？ ヨーロッパなどは使ってると思うのだが。
